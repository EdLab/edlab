Client side architecture: Players in the field

Sharath Gururaj

March 19, 2013

private please Hello, In this blog, I'd like to do a roundup of various tools and frameworks that are available for use on the client side (i.e., javascript) Since javascript as a language is very simple and offers very little structure, A bewildering array of open source tools and frameworks have sprung up, resulting in a veritable jungle. Quick distinction: Tool vs Framework: A tool offers a set of commo functionality, like jquery or underscore.js but it does not impose any opinions on how your code should be organized. Our code drives the tools. A framework on the other hand is opinionated, and forces us to do things in a particular way. Example Angular, Backbone etc. The framework drives our code. So! here is a brief roundup of the players of the field. Javascript alternative (!)

  • Coffeescript is an extension to javascript which compiles to javascript. It supports nice scala like functional programming constructs and is definitely more readable than java, but its an additional layer.
IDEs
  • Eclipse web page editor has a reasonable autocomplete and syntax checking mechanism for HTML and javascript. which along with a chrome debug plugin is quite complete (see below)
  • Webstorm: The super awesome html/webpage editor which is the IDE for js just like eclipse is the IDE for java.
Utility tools:
  • Jquery: Needs no introduction. Its power lies in its rich selector methods and its cross browser support.
  • Underscore: Super boosts the functional programming nature of javascript. with a collection of 80 odd methods like forEach, map, reduce etc
  • Prototye.js Uses prototypal inheritance to simulate Class like behaviour in javascript.
  • RequireJS: The newest and most interesting. Implements an asynchronous module definition version of the commonjs "require" specification. Allows us to split source code into multiple files and lazily fetch the files when it is required.
Frameworks:
  • Angular.js The newest kid on the block, created by the head of testability team in Google, so testable from the ground up. Also does some neat tricks with watching changes in 'scope' variables and updating views automatically. But it has a very steep learning curve. Buzz words: Dependency injection, run-time compilation (lol), directives, deferred Promises etc. Its a different paradigm of programming that we need to get used to. It may seem all magic at first, or we would go "why the heck would someone do it that way"
  • Backbone.js uses a traditional MVC architecture and is very opinionated about frontend models being backed by err... backend models, which means that writing anything that deviates from REST verbs can be painful.
  • One other player, knockout.js is worth mentioning because it is lightweight and also uses live binding of models to views (the so called MVVM pattern)
Anciliary tools: These tools will not be downloaded to the client but run on node.js, helps in organizing our javascript code
  • Which brings us to our very first too: Node.js, the server side implementation of javascript.
  • Grunt.js A javascript test runner/"build" tool. It is to node.js in the same way that ANT is to java or make is to c++. It automates task like minification, concatenation and linting
Test frameworks.
  • Jasmine: a BDD style (yippee another buzzword) testing framework. Seems to be pretty popular with test cases being more readable than conventional unit testing frameworks
  • Qunit: the super simple unit testing framework built to test jquery. It is conventional unit testing method. Well supported.
  • mocha.js Well integrated with node.js so enables us to run fast unit tests in node. But since Node uses the V8 engine, the same engine that powers chrome, so speed advantages are not significant. More importantly, it supports different styles of testing. For example, both unit testing and BDD style testing. In summary, a big framework which needs to be learnt in order to take full advantage of.
  • Testacular.js The test runner with balls. (its their catchphrase). It spawns browser processes and autorealoads all tests in browsers so you have realtime feedback about failing tests. It is meant to replace the older JsTestDriver. Created by the angular team
  • Chrome dev plugin: A plugin for eclipse to connect to a chrome debugger instance for true javascript debugging, a.k.a JPDA for java
SO my recommendation is as follows:
  • Node.js for development and running javascript on console
  • grunt .js for minification, Linting etc
  • testacular for live test running.
  • angular js for a web framework.
  • jasmine for a test framework
  • requirejs for dependency management
  • jquery for general purpose tooling
  • Chrome debugger in eclipse (Strong plug here for buying commercial license for webstorm)
Is there other tools that are interesting or can be used by us? What do you think of the above setup.
