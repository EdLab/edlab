A Few Technical Notes on the UFR Alpha Launch

Eric Buth

December 23, 2010

HTML5 The new theme is written as HTML5. The validation spec is something of a moving target, but we've made every effort to hit the provisional requirements. Our primary reason for selecting HTML5 was not, as it sometimes is, the video and audio capabilities of the language but rather its semantic features. With it we've been able to more easily let the markup function as a data organization tool rather than as a method for specifying style and interactivity. This paradigm has positive implications for accessibility, syndication, and ease of coding further down the line. WordPress With this project we are still working with WordPress as a base platform, but we are making strides towards abstracting it's functionality to fit our own ideas about web publishing. To put it simply: we are waging war against the dashboard. Out of the box, WordPress (and every other publishing platform I can think of) depends heavily on a stylistically and functionally distinct “back-end” interface that is used to control the input of content viewed on the “front-end.” From our point of view, this splits the site in two and creates a host of new complications that work against our efforts at user involvement and understanding. In the UFR alpha release, all user management and content creation is controlled via panels and shadow boxes that are integrated directly with the front-end site. We selectively expose the underlying functionality of WordPress when and where it's appropriate without troubling the user with an “all-options” dashboard. Along those lines, we've also completely removed the concept of “commenting,” favoring instead a system where everything is a post. We've added the ability for posts to have parents and for parent posts to specify whether or not they will allow responses. The responses in this system have all the functionality of posts while they can also function as comments in certain contexts. Google Charts API Another user-visible feature that was important to us in this release was creating easy to understand metrics to help both teachers and students understand how they are impacting the site. The Google Charts API has been a big help in visualizing some of this information in that it requires us to write minimal code ourselves while still iterating through a pretty wide set of possibilities.
