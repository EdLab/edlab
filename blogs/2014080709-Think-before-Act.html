Think before Act

Luming Nie

August 7, 2014

For the past three weeks, I've focused mainly on modifying some backend code for calculating databases and writing test cases. During this process, one thing I've learned is that to be a programmer, I should "think before act". I got this feeling basically because I was trapped by some undesired problems derived from my rude coding as well as "copy and paste". Sometimes, I even feel like what I am doing is not "Object-Oriented Programming (OOP)", but "Stack Overflow Oriented Programming (SOP)". This finally led to errors that cost me more time to debug.

After getting stuck and slowly moving forward for about two weeks, I tried to improve this situation. Now, every time I add new features or modify existing code, I definitely think one or two more steps further on "any optimization?", "project structure organization?" and also "test results with fake data?" etc. All this thinking before acting saves programmers more potential debugging and optimizing time. Meanwhile, communicating and asking also play essential roles in project development. I really made some mistakes in Git version control system that cost me much more time to fix than to ask colleagues before pushing or merging code.

Anyway, in computer science, it seems you should learn by doing. Hands-on experience gives us chances to make mistakes, and solving these mistakes let us gain real knowledge (BTW, I think that's why interactive programming courses like Codecademy are more acceptable and popular).
