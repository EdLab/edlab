What features would you like in a client side architecture

Sharath Gururaj

March 18, 2013

private please In this blog, I'd like to discuss the features that makes a good architecture for client side code(i.e., the code that runs in the browser).

  • Whatever framework we use should not be verbose. i.e., there should be very little boilerplate code that would need to be written to achieve a functionality. One good measure of this is, how much code do we need to write a standard functionality, say the login screen. We can use this as one criterion to evaluate different ways of organizing code.
  • all javascript code should be well commented and linted according to an agreed upon style guide. It is worth the effort to configure our respective IDEs to do this automatically.
  • It should be possible to organize source code into separate files for better readability. The separation would have a twofold advantage:
    • Given a functionality on the UI, for example a button press, It should be possible immediately, due to convention, to tell what files are involved. i.e., where is the source file for the button code, which source file has the handler for the button click etc.
    • Each file would be testable separately.
  • All dependencies and paths should be normalized. i.e., It should not change from machine to machine, or even from development mode to production mode.
There should be two "modes" of running the code: a development mode and a production mode.
  • The behaviour of development and production code should be identical
For development mode:
  1. It should be possible to develop in a full IDE like eclipse ec, with debugging cababilities (like breakpoints etc), without having to resort to in-browser debugging like firebug etc.
  2. Developers should be able to see the effects of changes they are making in realtime, either in the form of unit tests or the browser refreshing itself on every save
  3. One should be able to run all (or at least one particular) unit test on every save, so that developers have instant feedback on whether the test is breaking or not. This will allow true test driven development, which also makes up for a lack of compiler.
  4. The code should be organized such that no intermediate steps are required between changing the code and seeing the change on the browser. A browser refresh should be enough.
  5. In most cases, It should be possible to do "view source", save the source to disk and view the saved file in a browser without having to modify paths. i.e., all the image paths etc should "just work" when saved to disk.
  6. exceptions should show a correct stack trace with line numbers to original source, and should print out local variable information
Production mode
  • The code should be minified. Is it required to combine all files to a single file?
  • explore if it is possible to use "real" source lines for exceptions that might occur in production code.
  • explore if we can transmit client side exceptions to the server for later debugging. Please suggest other features/changes that would make for a truly pleasant and fast development experience.
