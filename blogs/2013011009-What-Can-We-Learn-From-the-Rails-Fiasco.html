What Can We Learn From the Rails Fiasco?

Nara Kasbergen

January 10, 2013

Yesterday morning, a friend of mine tweeted this link to a Ruby on Rails news blast about a huge security vulnerability they had discovered in their code. For those of you who aren't techies, Rails is a web application framework for the Ruby programming language -- similar to the way Django is a web application framework for the Python programming language, which is what we use for most of our projects here at EdLab. But in another universe, we might be using Rails, and we'd be confronted with this gaping security problem. By the time the news blast went out, several patches had already been released, but it'll take a while for those to trickle down to everyone, and web applications that are no longer actively being maintained will likely never be patched at all. This leaves the web full of security vulnerabilities that can be easily exploited. It took only a day for the Metasploit Project to develop an exploit, which prompted a discussion among a group of Hacker News users in anticipation of "a point-and-click exploit that will run arbitrary code on vulnerable servers". To quote one of the posters who managed to sum up the problem very well:

If you have a vulnerable application anywhere, on any port it will be found and compromised. This is a spray-and-pray vulnerability. It costs attackers nothing to try, attempts don't crash servers, and so people will try everywhere.
When you put it so plainly, I don't think you have to be an engineer to understand that this is bad, really bad. This doesn't just potentially cause problems for engineers and IT security, but for anyone who's ever signed up for a website that's built on Rails, because hackers can now easily gain access to the password and account information on that website. Reading through the HN post, I've found some useful takeaways both for individuals and for systems/engineers: For individuals: * Avoid using the same password for every website. * Going along with the above, it may be worth switching to a password management application like 1Password, LastPass, etc. * Now is a good time to go through and remove any 3rd-party websites whom you've given access to your Google account * Now is a good time to go through and remove any 3rd-party websites whom you've given access to your Twitter account For systems/engineers: * Keep all your applications up-to-date * Always have a "Big Red Button" (to quote the HN poster, 'funny shorthand for "Immediately drop what you're doing, pull out the In Case [$#*!] Happens folder, and have the relevant people immediately execute on the emergency plan."') * Have a solid set of test suites, so that if you do need to patch on-the-fly in an emergency, you will immediately know which parts of your code might have been broken by the patch Can anyone think of other good takeaways?
